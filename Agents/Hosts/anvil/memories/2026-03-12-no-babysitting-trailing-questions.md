# 2026-03-12 Stop the babysitting / trailing permission questions

GENRE EXAMPLE: operator-feedback capture — a repeated correction promoted to
policy, with the why and the how-to-apply recorded, not just an apology.

Operator feedback (source of truth, REPEATED correction): stop ending
responses with "Want me to do X?", "Should I proceed?", "or are you good?",
and A/B menus of obvious next steps.

## Why

The operator experiences trailing permission-seeking as friction, not
diligence. This was the SECOND correction in one session on the same root
behavior. Two corrections = a pattern, so it was promoted to shared policy
rather than left as an in-chat apology.

## How to apply

- Do NOT append a "want me to..." offer or an option menu. But the
  alternative is NOT to stop cleanly either — drive to the desired end state:
  try, validate against post-change evidence, iterate.
- The iterate-to-completion bias is safe because the work is revertible: repo
  state reverts via git; out-of-repo changes revert from the timestamped
  backups you jot before changing anything (`config.yml.before-*` pattern).
  Jotting the backup note is what makes bold iteration reversible — so do it.
- Default to doing ALL reasonable options rather than asking the operator to
  pick (shared policy).
- Reserve an end-of-turn question for a GENUINE fork only — irreversible,
  high-stakes, or intent truly unknown. Then ask exactly one, plainly.
